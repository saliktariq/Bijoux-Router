Bijoux Quota-Aware LLM Gateway The Complete Idiot's Guide — from zero to production

This guide walks you through everything you need to use the Bijoux Router, step by step. No assumptions about your experience level. If you can copy-paste, you can use this.

What Is This? (Plain English)

Imagine you have accounts with multiple AI services (like Google Gemini, OpenRouter, Groq, etc.). Each account has a daily limit on how much you can use it. When one runs out, you have to manually switch to another. Annoying, right?

Bijoux Router solves this. You write one line of code to send a request, and the system:

  1. Picks the best provider that still has quota left
  2. Sends your request to it
  3. If that provider says "nope, you're out of quota" — it automatically retries with the next one
  4. Tracks how many tokens you've used on each provider
  5. Returns one clean response — you never have to think about which provider answered
✓ The One-Sentence Summary
One API call in → best available provider selected → quota tracked → automatic failover if needed → clean response out.

How It Works (The Flow)

Your Code │ ▼ router.process(LLMRequest(prompt="Hello")) │ ▼ ┌──────────────────────────────────┐ │ 1. Estimate tokens needed │ │ 2. Pick best provider by quota │ │ 3. Reserve tokens │ │ 4. Send request │ │ 5a. SUCCESS → record usage │──→ Return response │ 5b. QUOTA ERROR → try next │ │ 5c. ALL FAIL → raise error │ └──────────────────────────────────┘ │ ▼ SQLite DB (remembers usage across restarts)

Installation

Prerequisites

  • Python 3.11 or newer — Download Python
  • A terminal / command prompt
  • An API key from at least one LLM provider

Step-by-Step Install

1

Get the code

If you received this as a folder, just keep it wherever you like.
If it's in a git repo:

git clone <repo-url> bijoux-router
cd bijoux-router
2

Install it as a Python package

Open a terminal inside the bijoux-router folder and run:

# Basic install
pip install -e .

# With test/dev tools included
pip install -e ".[dev]"
ⓘ What does -e . mean?
It installs the package in "editable" mode from the current folder. Any changes you make to the source code take effect immediately — no reinstall needed.
3

Verify the install

python -c "from bijoux_router import BijouxRouter; print('OK')"

If you see OK, you're good.

Using Bijoux in Your Own Project

You have two options:

Option A: Install from the folder (recommended)

Point pip at the bijoux-router folder from any other project:

# From your project folder, pointing to wherever bijoux-router lives:
pip install -e "C:\path\to\bijoux-router"

# Or on Mac/Linux:
pip install -e "/home/you/bijoux-router"

Now you can from bijoux_router import BijouxRouter in any Python file in that project.

Option B: Copy the package folder

Copy the entire bijoux_router/ folder into your project and put the dependencies in your own requirements.txt:

pydantic>=2.5
httpx>=0.25
pyyaml>=6.0
click>=8.1
tiktoken>=0.5
tenacity>=8.2

Then run pip install -r requirements.txt.

5-Minute Quickstart

This gets you from zero to a working request in 5 minutes.

1

Set your API key as an environment variable

$env:GEMINI_API_KEY = "your-key-here"
export GEMINI_API_KEY="your-key-here"
2

Create a config file

Create a file called my_config.yaml:

providers:
  - name: my-gemini
    enabled: true
    provider_type: gemini
    api_key: "${GEMINI_API_KEY}"
    default_model: gemini-1.5-flash
    priority: 1
    quota:
      token_limit: 1000000
      period_type: day
      period_value: 1

storage_path: my_app_state.db
3

Write a Python script

import asyncio
from bijoux_router import BijouxRouter, LLMRequest

async def main():
    # Load your config
    router = BijouxRouter.from_yaml("my_config.yaml")

    # Send a request — that's it!
    response = await router.process(
        LLMRequest(prompt="What is the capital of France?")
    )

    print(response.content)     # "The capital of France is Paris."
    print(response.provider_name)  # "my-gemini"

    router.close()

asyncio.run(main())
✓ That's it!
Three things: set env var, create YAML, call router.process(). Everything else is optional.

The Config File (YAML)

The config file is where you tell Bijoux about your LLM providers. It's written in YAML — a simple text format that uses indentation (like Python).

Minimal Config File

providers:
  - name: my-provider        # A name you choose (any string)
    provider_type: gemini     # The type of LLM service
    api_key: "${MY_API_KEY}"  # Your API key (from environment variable)
    default_model: gemini-1.5-flash  # Which model to use

That's the absolute minimum. Everything else has sensible defaults.

Full Config File Structure

# ─── Global Settings ──────────────────────────────
selection_strategy:
  strategy_type: priority_quota  # How to pick providers
  fairness_cursor: true          # Spread load across same-priority providers

default_timeout_seconds: 60      # HTTP timeout
max_failover_attempts: 5         # Max providers to try per request
storage_path: bijoux_state.db    # Where to save usage data
log_level: INFO                  # Logging verbosity
redact_secrets: true             # Hide API keys in logs

# ─── Providers ────────────────────────────────────
providers:
  - name: ...
    # (see provider sections below)

  - name: ...
    # (you can have as many as you want)

Environment Variables & Secrets

⚠ Never hard-code API keys in config files!
Use the ${VARIABLE_NAME} syntax. Bijoux will read the value from your system's environment variables at startup.

How it works:

# In your YAML config:
api_key: "${GEMINI_API_KEY}"

# Bijoux reads the environment variable GEMINI_API_KEY at startup
# and replaces it with the actual value.
# If the variable doesn't exist → clear error message at startup.

Setting Environment Variables

Windows (PowerShell) — temporary (current session only)

$env:GEMINI_API_KEY = "AIzaSy..."
$env:OPENROUTER_API_KEY = "sk-or-..."
$env:GROQ_API_KEY = "gsk_..."

Windows — permanent (survives reboots)

[System.Environment]::SetEnvironmentVariable("GEMINI_API_KEY", "AIzaSy...", "User")

Mac / Linux — temporary

export GEMINI_API_KEY="AIzaSy..."
export OPENROUTER_API_KEY="sk-or-..."
export GROQ_API_KEY="gsk_..."

Mac / Linux — permanent (add to ~/.bashrc or ~/.zshrc)

echo 'export GEMINI_API_KEY="AIzaSy..."' >> ~/.bashrc
source ~/.bashrc

In a .env file (with python-dotenv)

# .env file
GEMINI_API_KEY=AIzaSy...
OPENROUTER_API_KEY=sk-or-...
GROQ_API_KEY=gsk_...
# In your Python code, BEFORE loading the router:
from dotenv import load_dotenv
load_dotenv()  # loads .env into environment

from bijoux_router import BijouxRouter
router = BijouxRouter.from_yaml("config.yaml")

  1. Go to Google AI Studio → API Keys
  2. Click "Create API Key"
  3. Copy the key (starts with AIza...)
  4. Set it as an environment variable: GEMINI_API_KEY

3. Complete Gemini Config (Every Option Explained)

- name: gemini-free                    # YOUR name for this provider (anything you want)
  enabled: true                        # true = active, false = skip this provider
  provider_type: gemini                # MUST be "gemini" for Google's API

  # ─── Connection ──────────────────────
  base_url: "https://generativelanguage.googleapis.com/v1beta"
      # The Gemini API endpoint. You almost never need to change this.
      # Default: the above URL. You can omit this line entirely.

  api_key: "${GEMINI_API_KEY}"
      # Your API key. ALWAYS use ${ENV_VAR} syntax — never paste the key directly.

  timeout_seconds: 60
      # How long to wait for a response before giving up (seconds).
      # Gemini is usually fast, 60s is generous.

  # ─── Models ──────────────────────────
  default_model: gemini-1.5-flash
      # The model to use when the caller doesn't specify one.

  model_map:
      # Translation table: "what you ask for" → "what Gemini actually uses"
      # Lets your code use generic names like "gpt-4o-mini" and Bijoux maps
      # them to the right Gemini model automatically.
      gpt-4o-mini: gemini-1.5-flash
      gpt-4o: gemini-1.5-pro
      gpt-4: gemini-1.5-pro
      fast: gemini-2.0-flash
      # If you request a model NOT in this map, it's passed through as-is.

  # ─── Priority & Routing ──────────────
  priority: 1
      # Lower number = tried first. If you have 3 providers with priority
      # 1, 2, 3 — priority 1 is tried first, then 2, then 3.

  selection_weight: 1.0
      # For future weighted random selection. 1.0 is the default.

  # ─── Quota ───────────────────────────
  quota:
    token_limit: 1000000
        # Maximum tokens you want to use on this provider per period.
        # Set this to match your actual plan limit (or lower for safety).

    request_limit: 1500
        # Maximum number of requests per period. null/omit = unlimited.

    period_type: day
        # How long a quota period lasts.
        # Options: "minute", "hour", "day", "month", "custom"

    period_value: 1
        # Multiplier. period_type=day + period_value=1 = 1-day window.
        # period_type=hour + period_value=6 = 6-hour window.

    reset_mode: fixed
        # "fixed" = resets at calendar boundaries (midnight, top of hour, etc.)
        # "rolling" = sliding window from the first request

  # ─── Retry Policy ────────────────────
  retry_policy:
    max_retries: 2
        # How many times to retry THIS provider on temporary errors
        # (like 500 Internal Server Error). Does NOT count as failover.

    backoff_base: 1.5
        # Exponential backoff: wait 1.5s, then 3s, then 6s, etc.

    backoff_max: 15.0
        # Never wait longer than this many seconds between retries.

    retry_on_transient: true
        # true = retry on 5xx errors, timeouts, network errors.
        # false = fail immediately on any error.

  # ─── Cooldown Policy ─────────────────
  cooldown_policy:
    cooldown_seconds: 120
        # After too many failures, put this provider "on ice" for this long.

    failure_threshold: 3
        # How many consecutive failures trigger cooldown.

    quota_exhaustion_cooldown_seconds: 600
        # When provider says "quota exhausted", cool down for this long (10 min).
        # This prevents hammering a provider that's clearly out of quota.

  # ─── Extra Options ───────────────────
  headers: {}
      # Extra HTTP headers. Gemini doesn't need any. You can omit this.

  tags: [free-tier, gemini, google]
      # Labels for your own organization. Not used by the router.

  metadata: {}
      # Arbitrary data for your own use.

  failover_enabled: true
      # true = if this provider fails, try the next one.
      # false = if this provider fails, the entire request fails.

  continue_on_auth_error: false
      # Should the router try another provider if this one says "bad API key"?
      # Usually false — if your key is wrong, fix it, don't mask it.

  continue_on_invalid_request: false
      # Should the router try another provider on a 400 Bad Request?
      # Usually false — the request is probably bad on all providers.

  cost:
      # Optional. For future "cheapest provider" selection.
      input_cost_per_1k: 0.0
      output_cost_per_1k: 0.0
      currency: USD

Provider Setup: OpenRouter

1. Get Your API Key

  1. Go to openrouter.ai/keys
  2. Click "Create Key"
  3. Copy the key (starts with sk-or-...)
  4. Set it as an environment variable: OPENROUTER_API_KEY
ⓘ About OpenRouter
OpenRouter is a proxy that gives you access to many LLM models (GPT-4, Claude, Llama, etc.) through a single API. It uses the OpenAI-compatible format. Some models have free tiers.

2. OpenRouter Free Tier Limits

Free-tier limits vary by model. Check openrouter.ai/models and filter by "Free". Typical: 200 requests/day, limited tokens.

3. Complete OpenRouter Config

- name: openrouter-main
  enabled: true
  provider_type: openrouter          # MUST be "openrouter"

  # ─── Connection ──────────────────────
  base_url: "https://openrouter.ai/api/v1"
      # The OpenRouter API endpoint. This is the default — you can omit it.

  api_key: "${OPENROUTER_API_KEY}"
      # Your OpenRouter API key.

  timeout_seconds: 90
      # OpenRouter sometimes routes to slower providers, 90s is safer.

  # ─── Models ──────────────────────────
  default_model: openai/gpt-4o-mini
      # OpenRouter model names follow the format "provider/model-name".
      # Check openrouter.ai/models for the full list.

  model_map:
      gpt-4o-mini: openai/gpt-4o-mini
      gpt-4o: openai/gpt-4o
      claude-sonnet: anthropic/claude-3.5-sonnet
      llama-70b: meta-llama/llama-3.1-70b-instruct
      free-fast: meta-llama/llama-3.1-8b-instruct:free
      # Note the ":free" suffix for free-tier models on OpenRouter!

  # ─── Priority & Routing ──────────────
  priority: 2
      # Try this AFTER priority-1 providers (like a local model or Gemini free).

  # ─── Quota ───────────────────────────
  quota:
    token_limit: 500000
    request_limit: 1000
    period_type: day
    period_value: 1
    reset_mode: fixed

  # ─── Retry Policy ────────────────────
  retry_policy:
    max_retries: 2
    backoff_base: 1.0
    backoff_max: 10.0
    retry_on_transient: true

  # ─── Cooldown Policy ─────────────────
  cooldown_policy:
    cooldown_seconds: 60
    failure_threshold: 3
    quota_exhaustion_cooldown_seconds: 300

  # ─── OpenRouter-Specific Headers ─────
  headers:
    HTTP-Referer: "https://your-app-url.com"
        # OpenRouter requires a referer header. Set to your app's URL.
    X-Title: "My App Name"
        # Shows up in your OpenRouter dashboard for tracking.
    # Note: Authorization header is set automatically from api_key.

  tags: [openrouter, paid]
  failover_enabled: true
  continue_on_auth_error: false
  continue_on_invalid_request: false

Provider Setup: Anthropic (Claude)

Anthropic provides the Claude family of models. Bijoux has a dedicated adapter for the Anthropic Messages API because it uses a proprietary format (system messages are a top-level parameter, not a message role; auth uses x-api-key, not Bearer).

Step 1 — Get your API key

  1. Go to console.anthropic.com
  2. Create an account and add billing
  3. Go to API Keys → Create new key
  4. Copy the key (starts with sk-ant-)

Step 2 — Set environment variable

# Windows
set ANTHROPIC_API_KEY=sk-ant-your-key-here

# Linux / macOS
export ANTHROPIC_API_KEY=sk-ant-your-key-here

Step 3 — Config entry

- name: anthropic
  enabled: true
  provider_type: anthropic     # uses the dedicated Anthropic adapter
  base_url: https://api.anthropic.com
  api_key: "${ANTHROPIC_API_KEY}"
  default_model: claude-sonnet-4-20250514
  model_map:
    claude-sonnet: claude-sonnet-4-20250514
    claude-haiku: claude-3-5-haiku-20241022
    claude-opus: claude-3-opus-20240229
  priority: 2
  timeout_seconds: 90
  retry_policy:
    max_retries: 2
    backoff_base: 1.0
    backoff_max: 10.0
    retry_on_transient: true
  cooldown_policy:
    cooldown_seconds: 60
    failure_threshold: 3
    quota_exhaustion_cooldown_seconds: 300
  quota:
    token_limit: 300000
    period_type: day
    period_value: 1
    reset_mode: fixed
  tags: [anthropic, claude]
  failover_enabled: true
Tip: Anthropic's max_tokens is required — if you don't pass it in your request, the adapter defaults to 1024. Set it explicitly in your LLMRequest for best results.

Provider Setup: Cloudflare Workers AI

Cloudflare Workers AI runs models on Cloudflare's global edge network. Bijoux has a dedicated adapter because the native API uses account-scoped URLs and a different response wrapper.

Step 1 — Get your credentials

  1. Go to dash.cloudflare.com
  2. Navigate to AI → Workers AI
  3. Note your Account ID (visible in the URL or on the right sidebar)
  4. Go to My Profile → API Tokens → Create a token with Workers AI read permission

Step 2 — Set environment variables

set CLOUDFLARE_API_KEY=your-cf-api-token
set CLOUDFLARE_ACCOUNT_ID=your-account-id

Step 3 — Config entry

- name: cloudflare
  enabled: true
  provider_type: cloudflare    # uses the dedicated Cloudflare adapter
  base_url: https://api.cloudflare.com
  api_key: "${CLOUDFLARE_API_KEY}"
  default_model: "@cf/meta/llama-3.1-70b-instruct"
  model_map:
    llama-70b: "@cf/meta/llama-3.1-70b-instruct"
    llama-8b: "@cf/meta/llama-3.1-8b-instruct"
  priority: 4
  timeout_seconds: 60
  quota:
    token_limit: 300000
    period_type: day
    period_value: 1
    reset_mode: fixed
  metadata:
    account_id: "${CLOUDFLARE_ACCOUNT_ID}"   # REQUIRED for native API
  tags: [cloudflare, edge]
  failover_enabled: true
Alternative: Cloudflare also offers an OpenAI-compatible gateway at https://gateway.ai.cloudflare.com/v1/{account_id}/{gateway}/openai — you can use provider_type: openai_compatible with that base_url instead.

Step 1 — Get your API token

  1. Go to replicate.com/account/api-tokens
  2. Create a new token (starts with r8_)

Step 2 — Set environment variable

set REPLICATE_API_TOKEN=r8_your-token-here

- name: replicate
  enabled: true
  provider_type: replicate    # uses the dedicated Replicate adapter
  base_url: https://api.replicate.com
  api_key: "${REPLICATE_API_TOKEN}"
  default_model: meta/meta-llama-3-70b-instruct
  model_map:
    llama-70b: meta/meta-llama-3-70b-instruct

Error Handling

from bijoux_router import (
    BijouxRouter, LLMRequest,
    AllProvidersExhaustedError,
    NoViableProviderError,
    BijouxError,
)

router = BijouxRouter.from_yaml("config/providers.yaml")

try:
    response = await router.process(LLMRequest(prompt="Hello"))
    print(response.content)

except NoViableProviderError:
    # ALL providers are either disabled, in cooldown, or out of quota
    # BEFORE any request was even attempted.
    print("No providers available right now. Try again later.")

except AllProvidersExhaustedError as e:
    # Every provider was tried and they ALL failed.
    print(f"All providers failed! Attempts: {len(e.attempts)}")
    for attempt in e.attempts:
        print(f"  {attempt['provider_name']}: {attempt['error_category']}")

except BijouxError as e:
    # Catch-all for any Bijoux-specific error
    print(f"Bijoux error: {e}")

finally:
    router.close()

Quota System Explained

This is the heart of Bijoux. Here's how it works in plain English:

The Lifecycle of a Request

1

ESTIMATE — Before sending

Bijoux estimates how many tokens your request will use (based on character count). This is approximate — it's impossible to know exactly before the provider processes it.

2

RESERVE — Before sending

Those estimated tokens are "reserved" in the database. This prevents two concurrent requests from both thinking there's enough quota and then both getting rejected.

3

SEND — The actual API call

The request goes to the provider. The provider may accept it or reject it.

4a

RECONCILE — On success

The reservation is released and replaced with actual token usage reported by the provider. This corrects any estimation error.

4b

RELEASE — On failure

The reservation is released (nothing was actually used). The request moves to the next provider.

⚠ Important: Quota Tracking Is Approximate

Bijoux tracks its own estimate of how much quota you've used. This can drift from reality because:

  • Token estimates aren't perfect
  • Some providers don't report actual token usage
  • You might use the same API key in other tools

This is by design. The failover system catches the difference: if Bijoux thinks you have quota but the provider says you don't, it simply tries the next provider.

Failover Explained

Failover = "if one provider fails, automatically try the next one."

What Triggers Failover?

ErrorFailover?Why
QUOTA_EXHAUSTED✓ YesProvider is out of tokens — try another
RATE_LIMITED✓ YesToo many requests/second — try another
INSUFFICIENT_CREDIT✓ YesBilling issue — try another
BILLING_BLOCKED✓ YesPayment problem — try another
MODEL_UNAVAILABLE✓ YesMaybe another provider has it
TRANSIENT_ERROR (5xx)✓ YesServer glitch — retry then failover
NETWORK_ERROR✓ YesConnection issue — try another

INVALID_REQUEST✗ No*Bad request — fix your code

* Can be overridden per-provider with continue_on_auth_error: true or continue_on_invalid_request: true

The Failover Chain

Request arrives │ ▼ Provider A (priority 1) ──→ QUOTA_EXHAUSTED ──→ cooldown A │ ▼ Provider B (priority 2) ──→ RATE_LIMITED ──→ cooldown B │ ▼ Provider C (priority 10) ──→ SUCCESS! ──→ Return response (with failover_attempts showing A and B failed)

Cooldown Explained

When a provider fails, Bijoux can put it "on ice" to avoid hammering it:

Two Types of Cooldown

1. Quota Exhaustion Cooldown

When a provider says "you're out of quota" → cooldown for quota_exhaustion_cooldown_seconds

cooldown_policy:
  quota_exhaustion_cooldown_seconds: 600   # 10 minutes
  # After receiving QUOTA_EXHAUSTED, RATE_LIMITED, INSUFFICIENT_CREDIT, or BILLING_BLOCKED

2. Repeated Failure Cooldown

When a provider fails failure_threshold times in a row → cooldown for cooldown_seconds

cooldown_policy:
  failure_threshold: 3      # After 3 consecutive failures (any type)
  cooldown_seconds: 60      # Cool down for 1 minute

After the cooldown expires, the provider is automatically available again.

from bijoux_router.providers.factory import register_provider_type
from bijoux_router.exceptions.errors import ProviderErrorCategory
from bijoux_router.models.request_response import LLMRequest, LLMResponse

class MyCustomProvider(BaseProviderClient):
    async def send_request(self, request: LLMRequest) -> LLMResponse:
        # Your implementation here
        ...

    def classify_error(self, status_code, body):
        if status_code == 429:
            return ProviderErrorCategory.RATE_LIMITED
        if status_code == 401:
            return ProviderErrorCategory.AUTH_ERROR
        return ProviderErrorCategory.UNKNOWN_ERROR

    return router_instance.get_provider_status()

Troubleshooting

Common Problems

"Environment variable 'X' referenced in config is not set"

You have ${SOME_VAR} in your YAML but haven't set that environment variable. Set it in your terminal before running your script. See Environment Variables.

"No viable provider found for request"

All providers are either:

  • Disabled (enabled: false)
  • In cooldown (recently failed too many times)
  • Out of quota for the current period

Run bijoux show-provider-status to see what's going on. Try bijoux reset-provider-usage <name> to clear counters.

"All providers exhausted"

Every provider was tried and they all failed. Check the failover_attempts on the exception to see why each one failed. Common causes:

  • All API keys are wrong → check credentials
  • All providers genuinely out of quota → wait for reset
  • Network issues → check internet connection

"Configuration validation failed"

Your YAML file has structural issues. Common mistakes:

  • Indentation is wrong (YAML is whitespace-sensitive, use spaces not tabs)
  • Missing required fields (name, provider_type)
  • Duplicate provider names

Run bijoux validate-config for a detailed error message.

My provider is always being skipped

  • Check enabled: true
  • Check the provider isn't in cooldown: bijoux show-provider-status
  • Check quota isn't exhausted: bijoux show-quota
  • Reset if needed: bijoux reset-provider-usage <name>

I get ModuleNotFoundError: No module named 'bijoux_router'

  • Make sure you installed it: pip install -e . from the bijoux-router folder
  • Make sure you're using the same Python environment where you installed it
  • Try python -c "import bijoux_router" to verify

The database file is locked

SQLite has a single-writer limitation. If you're running multiple processes writing to the same .db file, they'll queue up. For high-concurrency scenarios, use separate DB files per process or implement a PostgreSQL storage backend.
